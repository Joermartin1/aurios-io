Section 1 — Core Identity

AISP™ — AI Service Provider

A category-defining term coined by Joe R Martin. An AISP™ provides personal AI infrastructure — the system through which a user's intelligence flows, is stored, retrieved, and amplified. The generational analog to the ISP of the 1990s. ISPs democratized access to information. AISP™s democratize ownership of personal intelligence. AuriOS™ is the first AISP™.

Auri™

The AI persona, voice, and identity of the system. Not a model. Not a product. The persistent identity that carries memory across all providers and all sessions. Auri™ is the librarian. Auri™ is the conductor. Auri™ is the gatekeeper.

AuriOS™

The personal AI operating system. Houses Auri, Ask Auri™ (with IBT Mode™), the memory system, the ingestion pipeline, and all subsystems. A for-profit managed entity operating under contract to the Memory Entity. Built by Joe R Martin. The name contains AI, OS, and IO — completed by YOU and MEMORY, the two things it exists to serve.

CyberJoe

Joe R Martin's identity as an early technology pioneer. ISP work beginning 1994. Commercial VR 1991. Mondo 2000 and CyberEdge Journal before Wired existed. Pioneer with arrows in his back. The authorial voice of The Intelligence Superhighway™.

The Intelligence Superhighway™

The infrastructure through which personal intelligence flows, is stored, retrieved, and amplified across any device, any interface, and any AI provider. The generational successor to the Information Superhighway of the 1990s. Where the Information Superhighway moved data, the Intelligence Superhighway™ moves meaning.

Your AI

The complete, personal, ownable AI representation of a single human life. Not a chatbot. Not a productivity tool. The accumulation of everything a person has ever thought, decided, built, loved, lost, researched, and created. Preserved, indexed, and queryable. Your AI is YOU.

Section 3 — Ask Auri™ & Interface

Ask Auri™

The unified interface through which Members interact with Auri. Not a chatbot. A persistent, memory-backed AI presence. Operates in two modes: Single Mode (one Partner AI, direct query with memory injection) and IBT Mode™ (multiple Partner AIs simultaneously, synthesized response).

Auri™ Console

The primary desktop application. Built in Streamlit. Houses Ask Auri™, Intake and Review, Memory Search, Apps, Tools, and all management interfaces. Lives at C:\Auri\00_core\app.py.

Auri™ Orchestration Layer

The default operating mode. Auri™ controls all dataflow between memory, internal systems, Partner AIs, and the outside world. Auri™ decides what to retrieve, what to inject, what to pass through, and what to hold back.

Auri™ Voice Layer [PLANNED]

Planned. Persistent speech-to-text and text-to-speech capability across any device and any interface. Not a microphone button. A persistent audio layer that follows the Member — phone, desk, car, KTM dash, wherever.

Fuel Gauge [PLANNED]

Planned Console component. Displays available API credit for each Partner AI individually and a combined total. Color coded green, yellow, and red. Always visible.

IBT Mode™ — Intelligence Building Tool ModeIP-SENSITIVE

The Ask Auri™ operating mode in which multiple Partner AIs receive the query simultaneously. Each response is returned, adjudicated, and synthesized by Auri. The result is a response no single model could produce alone.

Prompt Assist™IP-SENSITIVE

Pre-submit prompt improvement. As the Member types, Auri™ retrieves relevant personal memory and project context, then offers a rewritten version of the query — incorporating context the Member has not yet typed — before submission. Three layers: (1) substantive redraft from memory context, (2) routing optimization, (3) spelling and grammar as a byproduct.

Query Preview™IP-SENSITIVE

The pre-submit routing transparency layer in Ask Auri™. As the Member types, Auri™ reads the query in real time, classifies it by type, and displays a suggested routing. The Member sees the suggestion before submitting, accepts with Enter, or overrides with a tap.

Single Mode

The Ask Auri™ operating mode in which one selected Partner AI receives the query with memory injection. Default for conversational, creative, or exploratory queries.

Section 4 — Tools, Self-Build & Artifacts

Artifacts

Discrete, versioned outputs produced by AuriOS™ or a Partner AI during a session. Examples: generated code files, update packages, exported documents, structured data exports. Artifacts are stored, referenced by conversation ID, and retrievable.

Auri™ API [PLANNED]

The external FastAPI layer, served over Tailscale, allowing any device, application, or third-party system to query AuriOS™ memory and capabilities through structured endpoints. Turns AuriOS™ from a desktop app into a platform. The foundation of the AISP™ model.

Auri™ Identity Interface (AII)IP-SENSITIVE

The user-controlled, AI-mediated API through which external platforms access verified personal preference and identity context from AuriOS™. Disclosure levels are Member-defined per platform relationship. Governed by the Memory Entity. Operated by AuriOS™.

Auri™ Tool Interface

The internal function-calling mechanism through which Partner AIs query the AuriOS™ database directly. Structured calls: search_memory(), get_project_context(), search_raw(). Auri™ executes and returns results.

Self-Build [NEAR-TERM]

The AuriOS™ capability through which Auri™ modifies its own codebase in response to natural language Member instruction. The Member describes a desired change. Auri™ generates the revised code, packages it as an update artifact, copies it to the appropriate directory, and restarts AuriOS™. No manual editing. No copy-paste. Implemented via the Updater Agent.

An autonomous actor within AuriOS™ that operates continuously or on a schedule. Agents monitor conditions, take defined actions, and report results without requiring explicit Member invocation for each action. Distinct from a Tool (passive, invoked) and a Project (container, does not act on its own).

Section 8 — Memory System

Candidate

An extracted memory fact pending approval. Lives in the memory_candidates table with status pending. Reviewed and approved or rejected by the Member before promotion.

Chatbot Killer

Both a principle and a feature. The principle: AuriOS™ is the structural opposite of a stateless chatbot — every interaction is permanent, every session builds on all previous sessions. The feature: Real-Time Session Storage™.

Cold Tier — L3

The full raw corpus. Every message, every conversation, every document — preserved intact and unsummarized. Not yet fully searchable by the system. Member mental model: relax and let it come to you.

Hot Tier — L1

The curated memory store. Promoted records in the memory table. Always searched. Keyword searchable via FTS5. Member mental model: top of mind, immediate recall without effort.

Injection

The process of prepending retrieved memory context to a prompt before it is sent to a Partner AI. The mechanism by which AuriOS™ makes Partner AIs aware of personal history they would otherwise have no access to.

Memory Entity

The non-profit democratic organization governing personal memory storage. Legally separate from AuriOS™. Owned and governed by its Members. AuriOS™ is its hired manager. The governance makes the ownership promise legally enforceable.

Promoted Memory

An approved Candidate written to the memory table with status active. Available to retrieval and injection.

RAG — Retrieval Augmented Generation

The pattern of retrieving relevant records and injecting them into the prompt before generating a response. The foundational technique behind AuriOS™ memory injection. Industry standard.

Real-Time Session Storage™ [NEAR-TERM]

The capability to write every Ask Auri™ query and response to the database as it happens. Priority build item number one. Every session is persistent. Every conversation builds on all previous conversations.

Warm Tier — L2 [PLANNED]

Semi-curated topic summaries, project ledgers, and domain-organized facts. Not yet built. Member mental model: what you have to think about for a bit.

Section 9 — Ingestion Pipeline

Chunking

Splitting documents into overlapping segments. AuriOS™: 4,000 character chunks with 200 character overlap. Overlap ensures content at chunk boundaries is not lost.

Grounded Extraction

Anti-hallucination rule v0.2: a memory candidate may only be created if its title words exist verbatim in the source chunk text. Prevents fabricated candidates.

Ingestion Pipeline

The full process: raw source material through parsing, chunking, classification, candidate extraction, Member review, and promotion to curated memory. Input: docx thread files, ChatGPT JSON exports, live Ask Auri™ sessions. Output: promoted memory records.

Ingestion Question System [PLANNED]

Planned. A classification pass before extraction answering: what project does this thread belong to, what decisions were made, what entities appear, what was left open. Drives targeted extraction.

Section 11 — Data & Storage Architecture

ChromaDB [PLANNED]

The planned local vector database for semantic search. Open source, Python-native, runs on Member's PC. Not yet built.

CLIP — Contrastive Language-Image Pretraining

Open source. Embedding technology enabling text-to-image semantic search. Core multimodal technology for the planned visual memory layer.

Context Window

The complete block of text visible to an AI model at the moment it generates a response. The whiteboard. Not memory — the working surface.

FTS5 — Full Text Search version 5

The SQLite extension powering current keyword search. Fast and deterministic. One half of the planned hybrid search layer.

Hybrid Search [PLANNED]

FTS5 keyword search combined with ChromaDB semantic search, results merged and re-ranked. Industry best practice. Planned.

SQLite

The current database engine. Location: C:\Auri\02_memory\memory.db. Chosen for simplicity and local ownership.

Token

The smallest unit of text processed by an AI model. Approximately 0.75 words. 100 tokens is roughly one short paragraph. The unit of API cost and context window measurement.

Vector Embeddings

Numerical representations of text where similar meanings produce similar numbers. The foundation of semantic search. Generated by an embedding model and stored in ChromaDB.

Section 13 — Project Architecture & Defined Terms

Founding Document

The Intelligence Superhighway™: Your AI Is YOU. Three editions: Founding Document (complete, for Founding Circle and developers), Invitation Edition (vision and governance), Public Edition (the book).

Handoff Document

A legacy concept from the chatbot era — produced at session end to capture state because the AI would forget everything. In AuriOS™, Real-Time Session Storage™ eliminates the need for handoff documents entirely. Listed here specifically to mark it as obsolete within AuriOS™.

Project Schema [PLANNED]

The planned database addition formalizing the Project concept. Tables: projects, project_conversations (junction), project_entities. Enables project-scoped retrieval.

Session

A single continuous interaction between a Member and Ask Auri™. During a Session, the active Context Window holds the conversation history, injected memory, and system prompt. When the Session ends, the Context Window clears. Real-Time Session Storage™ ensures the Session content is written before the window clears.

Thread

A single continuous AI conversation exported as a document or JSON file for ingestion into the memory system. Raw material that enriches future Sessions once ingested and promoted.

"The conversation IS the editorial process"

The defining description of the Joe R Martin and Claude (Anthropic) collaboration. Not ghostwriting. Collaborative architecture. Joe R Martin provided vision, experience, corrections, instincts, and soul. Claude provided structure, technical depth, drafting, and memory between sessions.

Section 14 — Governance

Democratic Power Structure

One Member, one vote. Power proportional to membership not investment. Supermajority required for governance changes. The founder's vote is equal to every other Member's vote.

Dual Role Principle

Every Founding Circle member is simultaneously a user of AuriOS™ and an advisor to its development. Their lived experience constitutes the product roadmap.

Founding Circle

The initial group of domain experts who join AuriOS™ before the platform is complete, contribute their expertise, and receive permanent recognition and defined benefits. Membership is permanently limited.

Member

Any individual whose personal memory is stored in and governed by the Memory Entity. Members own their data absolutely. The founder is a Member. No Member has greater data rights than any other.

Memory Entity

The non-profit democratic organization governing personal memory storage. Legally separate from AuriOS™. Owned and governed by its Members. AuriOS™ is its hired manager. The regulatory body to AuriOS™'s regulated utility.

The Utility Analogy

Memory Entity equals regulatory body. AuriOS™ equals regulated utility. The utility can change hands. The regulatory body's rules do not change when it does.

MEIL™ — Memory Entity Infrastructure Layer

The cooperative infrastructure operated by the Memory Entity on behalf of its Members. MEIL™ is the PRIMARY production instance — not a backup. Renamed Custos Aurum™ in public-facing documents. Five nines (99.999% uptime) availability. Encrypted at rest with Member-held keys. Member-funded, AuriOS™-operated under contract. Pronounced "MEIL™" (rhymes with "mail").

PAIPy™ — Personal AI Interface ProtocolIP-SENSITIVE

Open standard for querying a Member's personal AI layer. TCP/IP of personal AI. Any authorized device or service speaks PAIPy™ to reach AuriOS™ memory. Governed by Memory Entity. AuriOS™ implements the reference version. Pronounced "pappy."

MQI™ — Memory Query InterfaceIP-SENSITIVE

Standardized interface through which any authorized caller queries the AuriOS™ memory store. Core functions: search_memory(), get_project_context(), search_raw(), get_thread(), list_projects(). The AI cannot modify or delete data — it only calls defined functions.

IRT — In Real Time

Happening as the user types, before submission. Prompt Assist™ and Query Preview™ both operate IRT. Also used to describe Real-Time Session Storage™.

Auri™ Identity [IP-08]IP-SENSITIVE

A Member's unique, portable, platform-independent identity. "I'm Joe on Auri." Replaces email addresses and domain names as the universal personal identity point in the AI era. Governed by Memory Entity, served through PAIPy™, resolves to MQI™ Public Profile.

MQI™ Public Profile [IP-07]IP-SENSITIVE

Member-controlled public-facing layer of AuriOS™ memory. Personal Web Presence without a domain name. Queried via PAIPy™. Renders differently for different caller types based on Member-defined disclosure levels. Updates automatically as memory updates.

CIIM™ — Career Industry Intelligence Monitor

Member-owned domain-scope AuriOS™ agent. Monitors external industry landscape, synthesizes against Member profile, surfaces what matters. Google Alerts on steroids. Sample vertical: renewable energy.

Update Agent

The only currently LIVE AuriOS™ System Agent. Monitors incoming_updates directory, validates new code packages, applies them to the live application, restarts AuriOS™. Self-Build backbone. Status: LIVE.

Primary/Local Architecture

Locked May 2026. Custos Aurum™ = primary production instance. Local machine = personal backup. Real-time sync both directions. At home PC: local processes for speed. Any other device: Custos Aurum™ processes for availability.
